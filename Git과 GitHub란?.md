# Git & GitHub란?




# 1. Git

  - 분산형 버전 관리 시스템이다.
    
  - 오픈 소스이다.
    
  - 로컬에서 대부분의 작업을 처리하기 때문에 속도가 빠르다.
    
  - 브랜치를 매우 가볍게 생성하고 관리할 수 있다.
    
  - 형상 관리 도구 중 하나이다.
    
  - 내부적으로 보면 파일의 변화 기록을 저장하는 데이터 베이스다.
    
  - 매 커밋마다 프로젝트 전체를 스냅샷처럼 저장한다.
    
  - 여러 사람이 작업할 때 협업 기록을 관리해준다.
    
  - SHA-1 Hash를 사용하여 데이터 무결성을 보장한다.
    
>SHA-1 Hash : 데이터에 고유한 해시 값을 부여하여 데이터가 변조되었는지 확인할 수 있는 방법이다.

# 2. GitHub

  - Git이라는 분산 버전 관리 시스템을 기반으로 하는 웹 기반 플랫폼이다.
>그래서 Git의 개념을 이해하면 GitHub의 기능도 함께 이해할 수 있는 부분이 많다.

  - 언제, 어디서나 협업 프로젝트를 쉽게 진행할 수 있도록 돕는 역할을 한다.
    
  - 코드 변경을 리뷰하고 토론하는 기능을 제공한다.
    
  - 이슈 관리, 프로젝트 관리, CI/CD 자동화 같은 개발 지원 기능을 제공한다.
    
  - 다른 개발자의 프로젝트를 포크해서 가져와 수정하고 기여가 가능하다.
    
  - 팀원들이 특정 코드 라인에 댓글을 달아 토론 할 수 있다.
    
  - 프로젝트 결과를 다른 사람에게 보여주고 공유하는 서버 역할을 한다.
    


# 2.1 Git Flow
  - 프로젝트의 소스코드를 관리하기 위한 브랜치 관리 전략이다.<br>
  
  - 여러 개발자가 협업하는 대규모 프로젝트에서 코드 엉킴을 방지하는 규칙이다.<br>
  
  - 역할에 따라 5가지 브랜치(Main, Develop, Feature, Release, Hotfix)를 나누어 사용한다.
>Main : 사용자에게 실제 서비스되는 완성된 코드만 관리한다.<br>
>Develop : 다음 배포를 위해 기능을 모으는 중심 브랜치이다.<br>
>Feature : 각자 맡은 새로운 기능을 개발하는 임시 브랜치이다.<br>
>Release : 배포 전 최종 테스트를 진행하고 버그를 수정하는 브랜치이다.<br>
>Hotfix : 배포된 서비스(main)에서 긴급한 오류가 발생했을 때 즉시 수정하는 브랜치이다.


 # 3. Git & GitHub 주요 용어
   
  **1.Repository:** 저장소 / 말 그대로 파일이나 폴더를 저장해두는 곳이다. <br>
  
 **2.Commit:** 스냅샷 / 파일을 추가, 변경한 내용을 저장소에 영구적으로 기록하는 스냅샷이다.<br>
 
 **3.Push:** 클라우드 업로드 / 추가, 변경한 내용을 원격 저장소에 업로드하는 작업이다.<br>

  **4. Merge:** 병합 / 서로 다른 Branch에서 작업한 내용을 하나로 합치는 작업이다.<br>

  **5.Fetch:** 변경 가져오기 / 원격 저장소 변경 내용만 가져오고 자동으로 병합은 하지 않는다.<br>

  **6.Pull**: 최신화 업데이트 /  GitHub의 최신 내용을 내 컴퓨터로 다운로드하고 자동으로 병합하는 작업이다.<br>
>Pull = Fetch + Merge
 
  **7.Staging Area:** 장바구니 / 커밋을 하기전,저장할 파일들을 골라 담아 놓는 대기 장소 이다.<br>
  
  **8.Clone:** 복제 / GitHub에 있는 원격 저장소를 내 컴퓨터로 그대로 복사 하여 가져오는 작업이다.<br>
  
  **9.Branch :** 가지 / 기존 코드에서 독립적으로 작업할 수 있도록 나누는 작업 공간이다.<br>
 

  # 3.1 Github에서만 있는 주요 용어

  **1.Fork:** 가져오기 / 다른사람의 GitHub 저장소를 내 GitHub 계정으로 복사하는 기능이다.<br>

  **2.Pull Request:** 요청 / 내가 수정한 코드를 원래 저장소에 반영해 달라고 요청하는 기능이다.<br>

  **3.Issue:** 기록 & 관리 / 프로젝트의 버그, 문제, 기능 요청 등을 기록하고 관리하는 기능이다.<br>
  
  **4.Star:** 즐겨찾기 / 마음에 드는 프로젝트를 즐겨찾기처럼 표시하는 기능이다.<br>
  
  **5.Watch:** 알림 / 특정 프로젝트의 업데이트 알림을 받는 기능이다.<br>
  
  **6.Github Actions:** 자동 실행 / 코드 테스트, 빌드, 배포 등을 자동으로 실행하는 CI/CD 기능이다.
  > CI/CD란? 코드를 올리면 자동으로 검사하고 자동으로 배포해주는것

# 4.Git vs GitHub의 공통점과 차이점 

**공통점**

  - 둘다 Git이라는 동일한 버전 관리 시스템을 기반으로 동작한다.
    
  - 코드의 변경 이력을 기록하고, 문제가 생겼을 때 과거로 되돌리는 버전 관리가 중요하다.
    
  - 브랜치를 나누고 합치는 동일한 메커니즘을 사용한다.
    
  - 여러 사람이 함께 프로젝트를 만들 때 코드 변경을 공유하고 관리할 수 있다.
    
  - 둘 다 텍스트로 된 소스코드를 관리하는데 최적화 되어있다.
    
  - 누구나 기술을 발전시키고 공유할 수 있도록 오픈소스를 사용한다.
    
  - Hash(해시)라는 고유 번호를 사용해 데이터가 깨지거나 조작되지 않았음을 보장한다.

**차이점**

| 구분 | Git | GitHub|
|----|---------|--------|
|종류|내 컴퓨터에 설치하는 소프트웨어|웹에서 접속하는 서비스/플랫폼|
|인터넷|오프라인에서도 작업 가능|반드시 인터넷 연결이 필요함|
|저장소| 내 로컬 PC| 원격 클라우드 서버|
|협업| 혼자서 기록을 관리함| 전세계 사람들과 코드를 공유하고 소통함|
|명령|주로 명령어를 직접 입력함(CLI)|클릭으로 조작 가능한 화면을 제공함(GUI)|
|보안|로컬 보안(사용자 책임)|클라우드 보안(플랫폼 관리)|
|비용|100% 무료 |부분 유료(저장 공간/고급 기능)|
>GUI: 마우스 방식<br>
>CLI: 타이핑 방식


# 5. 작업 시 주의 사항
  - "수정","소마고" 같은 메세지 대신 "플러터 1차시 과제"처럼 누가 봐도 변경 내용을 알수 있게 적어야한다.
   
  - 내가 코드를 올리기 전에 다른 사람이 올린 최신 코드가 있는지 먼저 확인하고 합쳐야 충돌을 방지할 수 있다.
    
  - API 키나 비밀번호 등 개인정보가 포함된 코드는 절대 GitHub에 올리지 않도록 주의한다.
    
  - Main 브랜치에 바로 코드를 짜는 습관은 위험하기에 새로운 브랜치를 만들어 작업하고 검토를 거쳐 합치는 습관을 기른다.

>1.main 브랜치는 언제든 사용자에게 보여줄 수 있는 완성본이여야 하기에 여기에 바로 코드를 짜다가 만약 에러가 난 상태로 끝내 버리면 다른 팀원들이 내 코드를 받았을 때 실행도 안되고 팀 작업이 올스톱 되는 민폐 상황이 생길수 있다.<br>
>2.main 에서 직접하면 코드를 빼고 싶을 때 특정 코드를 도려내기 어렵다.하지만 다른 브랜치를 쓰면 그 브랜치만 삭제하거나 합치지만 않으면 끝이다.
